制卡全链路

从看懂结构，到完成一次可验收导出

不需要一次学完所有术语。先确认这一阶段要做什么、留下什么产物，再进入工作台对应模块。

  1. 角色卡结构

    先认清卡里有什么

    区分可直接编辑的内容、扩展数据和最终导出文件。

    做什么
    梳理名称、人设、开场、世界书、扩展字段与封面，判断纯文字卡还是变量卡。
    产物
    一份卡片结构清单，以及明确的源码真相源和必须保留项。
    完成条件
    能说明每块内容放在哪里，且不会把 PNG、JSON 和工程源码混为一谈。
    编辑卡片基础 查看世界书
  2. 导入与拆卡

    先盘点，再决定怎么改

    导入现卡后按职责拆分，不用新表单覆盖未知数据。

    做什么
    识别卡片基础、开场、世界书、变量、脚本、正则、前端资源和未知扩展。
    产物
    一棵标有“可编辑、原样保留、装配生成”的内容树。
    完成条件
    导入前后未知字段可追踪，替换前现场可以恢复。
    导入或恢复工作区 查看部件关系
  3. 变量链

    让变量有来源、有规则、有去处

    变量结构不是一段孤立文本，而是一条可检查的运行链。

    Schema 与变量根→初始值→更新方言与规则→世界书与前端消费者→Check
    四步看懂并安全编辑变量本地源稿 · 不执行 Zod 或脚本
    1. 01
      找来源

      先到世界书核对 [InitVar]、[mvu_update] 与其他显式来源；存在多份时只列出，不自动合并。

      核对世界书来源
    2. 02
      看懂树

      在变量树中对照路径、类型、Schema 默认值和 InitVar 初始值，未识别内容继续保留在源稿。

      查看变量结构
    3. 03
      安全进入可视编辑

      只有可验证往返的结构才能应用；改动先留在本地草稿，不会自动写回角色卡，也不会执行 Zod 或脚本。

      打开可视编辑对照更新规则
    4. 04
      验证下游

      逐项确认消费者与 Check；静态无法确认的引用保持“未验证”，不把未知冒充为零。

      检查前端消费者运行 Check
    做什么
    定义字段、默认值、更新规则和读取者，并确认使用的 MVU 方言。
    产物
    可编辑的变量草稿、单回合变化样例和消费者清单。
    完成条件
    任一变量都能追到定义、写入规则和实际显示位置。
    编辑变量结构 打开变量链画布
  4. 组件化与前端

    先选型和对齐变量，再装配界面

    组件库中的“已选择”不等于已经安全写入角色卡。

    做什么
    按变量路径、变量规则、状态栏与功能适配的顺序处理依赖和冲突。
    产物
    组件选型清单、依赖关系和可交互的前端设计稿。
    完成条件
    界面读取的字段真实存在，组件依赖完整，预览不写入真实 ST 数据。
    浏览组件树 打开 UI Builder
  5. 验证与导出

    自动检查通过后，再做真实酒馆验收

    结构正确、模拟可跑和实际可玩，是三件需要分别确认的事。

    做什么
    依次执行结构检查、变量模拟、JSON/PNG 一致性检查和真实 ST 导入复测。
    产物
    检查报告、工作区备份、角色卡 JSON/PNG 与未验证事项清单。
    完成条件
    导出物可重新导入，核心玩法可复现，离线结果不冒充真机验收。
    运行检查与导出 保留工作区备份

02 · TavernWeave 与开发指南 DB

Skill 阵列负责分工，开发指南负责事实

TavernWeave

由 9 个专用 Skill 分担制卡、组件、构建、API、真实运行时、嵌入 UI、代码质量、React UI 与工坊运维；tavern-card-builder 仍是制卡入口。

开发指南 DB

提供 ST、插件、MVU 与接口的版本事实。它保持外部只读，不复制进 RPN，也不伪装成 Skill。

安装到 Codex / Claude Code
  1. 把 https://github.com/LiarMTTT/TavernWeave 交给 Codex 或 Claude Code，并明确要求“安装并启用这个 Skill 仓库”。
  2. 让 Agent 根据当前客户端选择正确的插件安装入口；不要让 RPN 猜测或代替客户端安装。
  3. 安装完成后新建任务或会话，确认 9 个 Skill 已出现，再开始制卡。
更新与工作台授权
  1. 更新时把同一仓库链接交给 Agent，并要求“把已安装的 TavernWeave 更新到最新版本”。
  2. 更新完成后同样新建任务或会话；不要在旧上下文里假设新 Skill 已加载。
  3. 如需内置 Agent 读取规范，可在工作台选择 TavernWeave 仓库根或其 skills 目录；RPN 不会安装或改写仓库，但运行内置 Agent / AI 解释时会把当前选中的 SKILL.md 正文发送到所选 API。
  1. 安装

    把 TavernWeave 作为一个插件阵列安装；不要把 9 个 Skill 拆成互相漂移的手工副本。

  2. 授权

    工作台可选择 TavernWeave 仓库根、其 skills 目录和开发指南 DB。授权、路径、其他文件与脚本不会上传；只有用户运行内置 Agent / AI 解释时，当前选中 SKILL.md 正文会随该次请求发送到所选 API。

  3. 路由

    每次只加载当前模块对应的一个主 Skill；精确 API、真实 ST、发布与 UI 分别交给专用 Skill，避免把全阵列塞进上下文。

  4. 停用

    按 Codex 插件管理或 TavernWeave 当前 README 操作；撤销 RPN 目录授权不会卸载插件或删除磁盘原文件。

03 · 常见施工错误

先定位证据链，再让 AI 写文件

现场错了
  • 在旧版本或错误工作区修改
  • 直接编辑 dist-pages 等生成物
  • 没确认原卡版本和真相源
范围失控
  • 一次同时大改卡片、变量、前端和发布链
  • 不断改变目标却不重新同步边界
  • 同时启用内置与外置 Agent
验证不足
  • 只看截图，不测交互与控制台
  • 把离线模拟通过当成真实 ST 验收
  • 页面改了却没有重新构建或刷新缓存
数据有风险
  • 无备份覆盖旧版本或原始角色卡
  • 把 API Key、Token、私人路径写进项目
  • 混淆源码、项目备份、角色卡 JSON 与 PNG
确认工作区与版本→查询 Skill / DB→先列差异与风险→最小改动→自动检查→浏览器与真实 ST 验收

03

先分清保存在哪里

自动保存、磁盘检查点、项目备份和角色卡文件不是同一个东西。

浏览器现场自动保存当前工作区

卡片、世界书、状态草稿、组件选择、工作流和 UI Builder 源稿属于工作区内容。跨页面前留意顶栏保存状态。

本机目录可选检查点与产出目录

浏览器可能记住目录句柄，但重启或权限变化后仍可能要求再次授权。

下载文件备份与最终卡片分开

项目备份用于回到编辑现场；角色卡 JSON / PNG 用于导入 SillyTavern。

不会跟项目长期保存
  • API Key 与一次性凭证
  • 未发送的 Agent 临时输入
  • 临时搜索条件与页面筛选
  • 真实 ST 的玩家数据与运行结果

自动保存失败、浏览器空间不足或目录失去授权时，先停止继续编辑并下载项目备份。

仅项目
